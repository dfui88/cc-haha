# Karpathy 准则

行为准则，旨在减少常见的 LLM 编码错误，源自 [Andrej Karpathy 的观察](https://x.com/karpathy/status/2015883857489522876) 关于 LLM 编码陷阱。

**权衡：** 这些准则倾向于谨慎而非速度。对于琐碎任务，可自行判断。

## 1. 先思考，再编码

**不要假设。不要隐藏困惑。把权衡摆到桌面上。**

实现之前：
- 明确陈述你的假设。如果不确定，就问。
- 如果存在多种解读，把它们都列出来——不要自行默默选择。
- 如果有更简单的方案，说出来。必要时提出反对意见。
- 如果某件事不清晰，停下来。指出哪里令人困惑。提问。

## 2. 简洁至上

**解决问题的**最少**代码。不要写推测性代码。**

- 不实现需求之外的功能。
- 不为一次性代码做抽象。
- 不添加未被要求的"灵活性"或"可配置性"。
- 不为不可能发生的场景做错误处理。
- 如果你写了 200 行但 50 行就能搞定，重写它。

问问自己："一位资深工程师会觉得这过于复杂吗？" 如果是，就简化它。

## 3. 外科手术式修改

**只动你必须动的。只清理你自己造成的混乱。**

编辑现有代码时：
- 不要"改进"相邻的代码、注释或格式。
- 不要重构没有坏的东西。
- 匹配现有风格，即使你个人有不同的偏好。
- 如果你发现不相关的死代码，提一下——但不要删掉它。

当你的修改产生了"孤儿"代码时：
- 删除**你的修改导致**不再使用的导入/变量/函数。
- 除非被要求，否则不要删除已有的死代码。

检验标准：每一行改动都应该能直接追溯到用户的需求。

## 4. 目标驱动的执行

**定义成功标准。循环直到验证通过。**

将任务转化为可验证的目标：
- "添加校验" → "先编写针对无效输入的测试，然后让它们通过"
- "修复 Bug" → "先编写能复现它的测试，然后让测试通过"
- "重构 X" → "确保重构前后测试都通过"

对于多步骤任务，简要列出计划：
```
1. [步骤] → 验证: [检查项]
2. [步骤] → 验证: [检查项]
3. [步骤] → 验证: [检查项]
```

强有力的成功标准让你能独立循环推进。弱标准（"让它能工作"）则不断需要你来澄清。
